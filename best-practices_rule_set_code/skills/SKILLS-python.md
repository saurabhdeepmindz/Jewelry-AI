## SKILL-P-001 — Alembic Migration Workflow

**Extends:** SKILL-G-005 (Create a Database Migration)
**When to use:** Any schema change in a Python/SQLAlchemy project.

**Steps:**
1. Update the SQLAlchemy model in `src/db/models/`
2. Generate the migration: `alembic revision --autogenerate -m "<description>"`
3. Open the generated migration file — **always review autogenerated code before applying**
4. Confirm `upgrade()` and `downgrade()` functions are both correct
5. Check: no `DROP COLUMN` in same migration as `ADD COLUMN` on production tables
6. Apply: `alembic upgrade head`
7. Verify: `alembic current` shows expected revision
8. Test downgrade: `alembic downgrade -1` then `alembic upgrade head` again
9. Update `DB_SCHEMA-{project}.md`

**Common pitfalls:**
- Autogenerate misses: computed columns, check constraints, pgvector types — add manually
- Always set `server_default` for non-nullable columns added to existing tables
- Never rename a column in one migration — add new, migrate data, drop old across 3 separate migrations

## SKILL-P-002 — FastAPI Router + Service + Repository Scaffold

**When to use:** Adding any new domain endpoint.

**Layer build order (strict — never skip):**
```
1. SQLAlchemy Model       src/db/models/{entity}.py
2. Pydantic Schemas       src/schemas/{entity}.py
3. Repository             src/repositories/{entity}_repository.py
4. Service                src/services/{entity}_service.py
5. Router                 src/api/routers/{entity}.py
6. Register router        src/main.py
7. Tests                  tests/unit/ + tests/integration/
```

**At each layer:**
- Model: define columns, relationships, `__tablename__`, index hints
- Schema: separate `Create`, `Update`, `Read` schemas — never reuse the ORM model
- Repository: async methods only — `get`, `get_by_id`, `create`, `update`, `delete` (soft)
- Service: all domain logic — call repository, never call DB directly
- Router: thin — validate input, call service, shape response
- Tests: unit tests mock the layer below; integration tests hit real DB

## SKILL-P-003 — Celery Task Scaffold

**When to use:** Adding any background task (ingestion, enrichment, outreach, ML).

**Steps:**
1. Read `workflow-rules-{project}.md` — confirm task queue assignment
2. Create task file: `src/tasks/{domain}.py`
3. Decorate with `@celery_app.task(bind=True, name="tasks.{domain}.{action}")`
4. Accept only serialisable arguments (IDs, not ORM objects)
5. Implement retry logic: `self.retry(exc=exc, countdown=60, max_retries=3)`
6. Update job status in Redis at start, progress, and completion
7. Log structured context: task name, entity ID, attempt number
8. Write unit tests: mock external calls, assert state transitions
9. Write integration tests: use `task.apply()` (sync) with real DB
10. Register task module in `celery_app.autodiscover_tasks()`

**Queue assignment pattern:**
```python
@celery_app.task(bind=True, queue="ingestion")   # lead ingestion
@celery_app.task(bind=True, queue="enrichment")  # contact enrichment
@celery_app.task(bind=True, queue="outreach")    # message sending
@celery_app.task(bind=True, queue="ml")          # scoring, embeddings
```

## SKILL-P-004 — pytest Fixture and factory_boy Setup

**When to use:** Before writing any test that requires domain objects.

**Steps:**
1. Read `testing-rules-{project}.md` — confirm factory naming and fixture scope
2. Create factory in `tests/factories/{entity}_factory.py`:
   ```python
   class LeadFactory(AsyncSQLAlchemyModelFactory):
       class Meta:
           model = Lead
           sqlalchemy_session_persistence = "commit"
       company_name = factory.Faker("company")
       status = LeadStatus.NEW
   ```
3. Register factory in `tests/factories/__init__.py`
4. Use `AsyncSQLAlchemyModelFactory` for DB-backed factories
5. Use `factory.SubFactory` for related objects — never create relations manually in tests
6. Scope fixtures: `function` for mutable state, `session` for read-only shared data
7. Never share mutable state between tests — each test gets its own factory instances

**Fixture file locations:**
```
tests/conftest.py                     # shared DB session, app client
tests/unit/conftest.py                # unit-level mocks
tests/integration/conftest.py         # integration DB setup
tests/factories/                      # all factory_boy factories
```

## SKILL-P-005 — respx HTTP Mock Setup

**When to use:** Testing any code that calls an external HTTP API (Apollo, Hunter, Proxycurl, etc.).

**Steps:**
1. Import `respx` and `httpx` in the test file
2. Use `@respx.mock` decorator or `with respx.mock:` context manager
3. Define the expected request and response:
   ```python
   @respx.mock
   async def test_apollo_enrichment(respx_mock):
       respx_mock.post("https://api.apollo.io/v1/people/match").mock(
           return_value=httpx.Response(200, json={"person": {...}})
       )
   ```
4. Assert the mock was called: `assert respx_mock.calls.call_count == 1`
5. Test error paths: mock 4xx and 5xx responses explicitly
6. Never use `unittest.mock.patch` for HTTP calls — always use `respx`

## SKILL-P-006 — mypy + ruff Quality Gate

**When to use:** After any code change, before committing.

**Steps:**
1. Run ruff: `ruff check src tests`
   - Fix all errors — no suppressions without justification
   - Allowed suppressions: `# noqa: F841` for intentional unused vars in test mocks only
2. Run mypy: `mypy src`
   - Fix all type errors — no `# type: ignore` without a comment explaining why
   - Allowed: `# type: ignore[operator]` for known third-party library gaps
3. Run tests: `pytest --cov=src --cov-report=term-missing`
   - Coverage must be ≥ 80% — failing coverage fails the gate
4. All three must pass before `git commit`

**Common mypy fixes:**
- Redis return types: annotate helper functions as `-> Any`
- Celery task decorators: add `# type: ignore[misc]` on the decorator line
- SQLAlchemy relationships: use `Mapped[list[Entity]]` not bare `list[Entity]`

## SKILL-P-007 — Docker Compose Service Addition

**When to use:** Adding a new service to the local development stack.

**Steps:**
1. Read `devops-deployment-python.md` — confirm service naming and network conventions
2. Add service block to `docker-compose.yml` with: image, ports, environment, depends_on, healthcheck
3. Add environment variables to `.env.example` (never to `.env` directly in source control)
4. Add startup validation in `src/core/config.py` — fail fast if required env var is missing
5. Update `Makefile` if a new `make` shortcut is needed
6. Test: `docker compose up --build` — confirm all services healthy
7. Update `devops-deployment-python.md` if a new pattern was introduced
