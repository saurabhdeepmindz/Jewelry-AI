Annotations — WF-006: Admin Panel

  • [A1] Tab 1 (Users): default tab on page load. Manages platform user accounts.
  • [A2] Tab 2 (API Keys): manages external service credentials. Stored encrypted in DB or env-managed.
  • [A3] username → POST /api/v1/admin/users body — must be unique, lowercase letters/dots/numbers only
  • [A4] email → validated as valid email address before submission
  • [A5] password → temporary password. User must change on first login. Min 12 chars. Never logged.
  • [A6] role → one of: admin, manager, rep. Maps to User.role enum.
  • [A7] Users table: GET /api/v1/admin/users → User[]. Fields: username, email, role, is_active, created_at, last_login_at.
  • [A8] Role badge colours: admin=black, manager=dark-grey, rep=light-grey
  • [A9] Admin cannot deactivate own account. Deactivate button disabled when row = logged-in user.
  • [A10] Deactivate: PATCH /api/v1/admin/users/{id} with {"is_active": false}. Soft delete — user record preserved.
  • [A11] Inactive users shown greyed out. Cannot login. Still appear in table for audit trail.
  • [A12] Reactivate: PATCH /api/v1/admin/users/{id} with {"is_active": true}
  • [A13] Deactivation confirmation dialog: shown inline below the table when Deactivate is clicked. Must explicitly confirm before API call.
  • [A14] API keys: GET /api/v1/admin/api-keys → service name, is_active, last_rotated_at. Never returns key value.
  • [A15] Key preview: show first 4 + "••••" + last 4 chars only. For identification. Never full plaintext.
  • [A16] Revoked keys: shown greyed with revocation timestamp. For audit trail. Cannot reactivate — must add new key.
  • [A17] Security reminder always shown on API Keys tab. Encourages key rotation.
